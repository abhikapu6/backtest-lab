# Backtesting Lab – Build Plan

Based on the PRD and left-sidebar design. Execute phases in order.

---

## Frontend structure (left sidebar)

- **Layout**: Left sidebar with app name + nav; main content area as route outlet.
- **Routes**:
  - `/` → redirect to `/backtest/new`
  - `/backtest/new` – Run Backtest (config form)
  - `/backtest/:id` – Results (KPIs, equity/drawdown charts, trades table)
  - `/history` – Backtest history list
- **Design system**: Theme tokens (colors, typography, spacing); core components: Button, Card, Input, Select, Toggle, Table, KpiCard. Chart library: Recharts or similar.

---

## Phase 0 – Project and architecture setup [DONE]

- **0.1** Monorepo with `frontend/` and `backend/`, npm workspaces, TypeScript in both.
- **0.2** Backend: Express 5, modular folders (`routes/`, `services/`, `engine/`, `db/`). Frontend: React 19 + Vite + React Router.
- **0.3** ESLint + Prettier, `.env` config, Vite proxy `/api` → backend.

**Deliverable**: `npm run dev` starts both; `GET /api/health` responds.

---

## Phase 1 – Data model and database layer

- **1.1** PostgreSQL schema:
  - `candles`: symbol, date/time, open, high, low, close, volume
  - `backtests`: id, created_at, strategy, symbols, date_range, params (JSON), cost_model (JSON), initial_capital, status
  - `backtest_metrics`: backtest_id (FK), total_return, cagr, sharpe, volatility, max_drawdown, win_rate, profit_factor, num_trades
  - `backtest_equity`: backtest_id (FK), date, equity, drawdown
  - `backtest_trades`: backtest_id (FK), symbol, entry_date, exit_date, entry_price, exit_price, pnl, return_pct
- **1.2** Migrations and DB access (repositories or ORM models). Seed OHLCV for SPY, QQQ, AAPL, MSFT, TSLA (daily).
- **1.3** Basic validation (candle continuity, no negative prices); document timezone and trading-day assumptions.

---

## Phase 2 – Backtest engine core

- **2.1** Core types: Bar, Position, Order, PortfolioState, Signal, CostModelConfig, StrategyConfig. Strategy interface: inputs (bar/window, position, params) → signal (long/flat) per symbol.
- **2.2** Simulation loop: load candles → each bar → get strategy signal → decide trades (one position per symbol, long-only) → execute at **next bar open** → apply cost model → update equity and positions. Support 1 then up to 3 symbols.
- **2.3** Cost model: fixed commission per trade; slippage in basis points; optional enable/disable.
- **2.4** Outputs: equity series, drawdown series, trade list, daily returns.

---

## Phase 3 – Strategy implementations (MVP library)

- **3.1** Parameter schemas: name, type, description, min/max/step, defaults.
- **3.2** Implement:
  - **SMA Crossover**: fastWindow, slowWindow; long when fast SMA > slow SMA.
  - **RSI Mean Reversion**: lookback, oversold, overbought; long when RSI < oversold, exit per threshold.
  - **Bollinger Bands**: window, numStdDevs; long on cross below lower band, exit on middle/upper.
- **3.3** Strategy registry: strategyId → name, description, parameter schema, strategy function.

---

## Phase 4 – Metrics and analytics

- **4.1** Daily and cumulative returns from equity.
- **4.2** Risk: annualized volatility (252 days), Sharpe (rf=0), max drawdown (and optionally duration).
- **4.3** Trade metrics: win rate, average win/loss, profit factor, num_trades.
- **4.4** After each run: compute and persist to `backtest_metrics`, `backtest_equity`, `backtest_trades`; store full config in `backtests` for reproducibility.

---

## Phase 5 – Backend API layer

- **5.1** Endpoints:
  - `POST /api/backtests` – run backtest (symbols, date range, strategy id, params, cost model, initial capital) → backtest id
  - `GET /api/backtests/:id` – metadata + metrics + config
  - `GET /api/backtests/:id/equity` – equity/drawdown series
  - `GET /api/backtests/:id/trades` – trade list
  - `GET /api/backtests` – paginated history
  - `GET /api/strategies` – list strategies and parameter schemas
- **5.2** Server-side validation: allowed symbols, date range within data, strategy param bounds, cost bounds. No user code execution; strategy from registry only.
- **5.3** Target: backtest < 2s for ~10 years daily; optional cache for identical configs.

---

## Phase 6 – Frontend shell and navigation

- **6.1** Left sidebar layout: sidebar + main content outlet.
- **6.2** Routes as above; basic theme and components (Button, Card, Input, Select, Table, KpiCard, etc.).

---

## Phase 7 – Run Backtest page

- **7.1** Form: asset universe (multi-select MVP symbols), date range, strategy select, dynamic strategy params, cost toggle + commission + slippage bps, initial capital (default 10,000).
- **7.2** Inline validation; strategy description; optional config summary.
- **7.3** Submit → `POST /api/backtests` → on success navigate to `/backtest/:id`; loading and error states.

---

## Phase 8 – Results dashboard page

- **8.1** Header: strategy, symbols, date range; KPI cards; equity and drawdown charts; trades table (entry/exit date, symbol, prices, PnL, return %).
- **8.2** Charts via Recharts (or similar) from `/equity` data.
- **8.3** Fetch metrics, equity, trades; loading, error, and not-found handling.

---

## Phase 9 – Backtest history

- **9.1** History page: table with run date, symbols, strategy, date range, total return, Sharpe, "View" → `/backtest/:id`.
- **9.2** Optional: "Clone config" → `/backtest/new` with prefilled state.
- **9.3** Backend pagination/filters; frontend filters (optional for MVP).

---

## Phase 10 – Hardening and validation

- **10.1** Performance: profile 3 symbols / 10 years; optimize engine and DB access.
- **10.2** Unit tests: strategies, engine (no lookahead, PnL), metrics. Integration test: run backtest → results.
- **10.3** UX: empty states, error boundaries, loading skeletons.
- **10.4** README: execution assumptions (next-bar open, long-only, full capital), cost model, how to reproduce a run.

---

## Phase 11 – Post-MVP hooks (design only)

- Engine/API ready for: short selling, variable position sizing (e.g. Kelly, fixed fractional).
- DB/UI ready for: benchmark series, strategy comparison (multiple equity curves).
- Keep "daily" abstract (generic time index) for future intraday.

---

## Success criteria (from PRD)

- Backtest completes in under 2 seconds for daily data < 10 years.
- Metrics correct and reproducible.
- UI clean and professional; codebase modular and extensible.
